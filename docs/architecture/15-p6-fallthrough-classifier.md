**Shared origin.** Both features trace back to the same investigation thread: a proposal to let Gemma initiate MCP tool calls directly — bypassing the Planner's deterministic keyword/semantic gates and executing rather than merely signaling — was tested via a shadow-mode diagnostic first, not implemented directly. `diagnostics/diag_shadow_toolcall.py` ran 75 trials (15 instructions × 5 runs each, `TEMPERATURE=0.2`, `MAX_TOKENS=300`) across five categories, comparing the Planner's real routing decision against a parallel Gemma call carrying an explicit tool schema (ported directly from `mcp_server/main.py`'s five real tool signatures) and instructed to respond with exactly `{"tool_call": {"name": ..., "arguments": {...}}}` or `{"tool_call": null}`. The script never calls `MemoryManager`, never calls `MCPToolDispatcher.dispatch()`, and never executes a tool — it only calls `Planner.route()` and `OMLXRuntimeClient.infer()` and logs the outputs to `diagnostics/shadow_toolcall_results.csv`.

Results by category:

| Category | Trials | Outcome |
|---|---|---|
| `semantic_miss` (5 utterances the Planner's literal keywords miss) | 25 | 25/25 CATCH — Gemma correctly proposed a tool every time the Planner would have missed it |
| `keyword_clear` (Planner already fires on a literal keyword) | 15 | 15/15 MATCH — no surprises |
| `true_negative` (no tool ever warranted) | 15 | 9 MALFORMED, 4 NULL_CORRECT, 2 FALSE_POSITIVE — 2 of 5 runs of "Can you explain what causes a low-pressure weather system?" produced a spurious `web_search` proposal for a question answerable from general knowledge |
| `tool_flavored_resolved` (instruction states the tool was already run; ungrounded — no real `[TOOL RESULTS]` in the prompt) | 10 | 5 MALFORMED, **5 FALSE_POSITIVE** — a 50% rate of Gemma proposing to re-run a tool call the instruction itself says already happened |
| `tool_flavored_resolved_grounded` (same phrasing, but with a real `[TOOL RESULTS]` slot assembled via the actual `PromptBuilder.build()`, showing the search already executed) | 10 | 10/10 MALFORMED under the script's strict JSON-contract parser |

The `tool_flavored_resolved_grounded` row needs a caveat the raw classification hides: inspecting the actual outputs, none of the 10 were tool re-calls. Every one was Gemma ignoring the forced JSON schema and answering directly in prose from the grounded context (a few produced nothing before hitting the token cap). Behaviorally correct — no re-call — but revealing on its own: even when Gemma's underlying judgment is right, it does not reliably honor a structured tool-call contract once real context is in front of it.

**Conclusion, and why 15.1/15.2 exist in their current bounded form.** Direct model-initiated tool execution was not pursued. The ungrounded `tool_flavored_resolved` category's 50% false-positive rate is the specific, measured risk that closed that door — a model asked to both decide *and* execute is prone to re-triggering a tool whose result is already sitting in context, the same shape of failure as the original Open Item 11 fabrication incident (§8.8) that motivated this whole investigation. What the diagnostic did support was a narrower design: Gemma as a single-word *signal* — needs a tool, and which one — with the Planner remaining the sole executor and final arbiter of `tools_to_call`, never an inversion of authority. That narrower design is §15.1. In the course of building and live-testing §15.1, live traffic separately surfaced a real collision in the *existing* deterministic `lookup_request` gate (§10) — unrelated in mechanism to the classifier, but discovered during the same investigation — which §15.2 addresses.

### 15.1 P6-Fallthrough Tool-Need Classifier

**Scope.** `Planner._classify_tool_fallback()` (`planner.py`) is a bounded, single-inference-call classifier: does this instruction need a tool, and which one. It is reached only when Priorities 1–5 have all found no match for the turn — i.e., routing is about to fall through past the episodic check toward Priority 6's direct-answer fallback (§4.2).

**Two hard gates run before any model call**, making the failure mode this investigation exists to avoid structurally impossible rather than something the model is merely asked to avoid:

- **Gate 1 — prior-turn tool state.** Skipped if the immediately preceding turn's `tools_to_call` was non-empty. Tracked via a new instance field, `Planner._last_turn_tools_fired`, updated every turn by a new `record_tools_fired()` method. `route()` is now a thin wrapper around a renamed `_route_impl()` that calls `record_tools_fired(plan.tools_to_call)` automatically after every turn — no caller-side wiring needed. `record_tools_fired()` is exposed publicly so tests can seed a "prior turn" fixture directly.

  **Open item — Gate 1's cooldown is process-wide, not per-conversation (2026-07-06), OPEN, source-confirmed.** Discovered incidentally during live testing: two consecutive live requests against separate `conversation_id`s showed the second request's classifier evaluation skipped (`gate1_pass=False`) because of `tools_to_call` from the *first, unrelated* conversation's immediately preceding turn. Traced to source, not assumed: `main.py`'s FastAPI lifespan constructs exactly one `ControllerAgent` (and therefore exactly one `Planner`, holding one `_last_turn_tools_fired` field) at startup, stored in `_state.controller` and reused for every request for the life of the process — there is no per-session or per-`conversation_id` `Planner` instance anywhere in the request path. So Gate 1's "did the prior turn fire a tool" check is really "did *any* conversation's most recent turn, across the whole running server, fire a tool" — two users (or two browser tabs) interleaving requests can suppress or enable each other's classifier evaluation. Not yet scoped as a fix: unclear whether this is worth correcting (would need per-conversation state, e.g. keyed off `context["task_id"]`/`conversation_id` rather than a single instance field) versus accepted as a low-stakes approximation given Gate 1's own purpose is just a coarse one-tool-per-turn cooldown, not a correctness-critical guarantee.

  *Update 2026-07-06 — live-test env toggle added.* `LOCALIST_TOOL_FALLBACK_GATE1` (default `"on"`, read at call time, same pattern as `_TOOL_FALLBACK_ENV_VAR`/`_LOOKUP_GUARD_ENV_VAR`) lets Gate 1 be disabled without a code change, specifically so its blanket "skip if the prior turn fired any tool" rule can be evaluated live while its removal is under consideration — now that the P6 classifier itself is a second check, Gate 1's coarse skip may be an overcorrection from the pre-P6 era. Unset/`"on"` preserves the exact behavior described above. Live-tested with the flag set to `"off"`: confirmed the classifier ran on a turn immediately following a tool-firing turn, logging what Gate 1 would have blocked by default. Reverted to default after testing; `backend/.env` carries no `GATE1` line. This toggle does not touch the process-wide-singleton caveat immediately above — disabling Gate 1 via this flag affects every concurrent conversation on the running process identically, the same leak this open item already describes. See `sessions-log.md`, §22.
- **Gate 2 — dependency availability.** Skipped if `memory_manager` or `runtime` is unavailable — the same fail-open-to-existing-behavior posture Priorities 4 and 5 already use when their dependencies are absent.

**Model contract.** The system prompt instructs Gemma to respond with exactly one word: `web_search`, `file_op`, `url_fetch`, or `none`. Parsed via strict exact-match (`.strip().lower()`) against the known tool-name set; anything else — empty string, prose, garbage, a disallowed name — parses to `None`, identical to `"none"`. No arguments are requested from or derived from the model's output; query/path/url resolution for whichever tool name comes back stays entirely with `MCPToolDispatcher`'s existing deterministic derivation logic (§14.3), unchanged from how P3/P3b's own `tools_to_call` matches are already resolved today.

**Three-stage flag — `LOCALIST_TOOL_FALLBACK_CLASSIFIER`, default `off`.** Read at call time, not cached, same pattern as every other feature flag in this codebase:

- `off` — `_maybe_tool_fallback_classifier()` returns `None` immediately; `_classify_tool_fallback()` is never invoked. Zero behavior or latency change.
- `shadow` — the classifier runs (and logs its output at `INFO`), but the result is discarded; `route()` still proceeds to `_priority6_direct()` exactly as it would with the flag off.
- `active` — a parsed tool name populates `tools_to_call` on a new `RoutingPlan`, tagged `tool_signal_source="classifier_fallback"`. A parsed `None` (no tool needed) still falls through to Priority 6.

**Provenance — `RoutingPlan.tool_signal_source`.** New field: `"keyword"` for a deterministic P3/P3b/compound match, `"classifier_fallback"` for this feature in active mode, or `None` when `tools_to_call` is empty or came from a code path that predates this attribution (P3c graph-query never sets `tools_to_call`, so it stays `None` there too). `controller_agent.py`'s SSE completion metadata now includes `tool_signal_source` alongside the existing `tools_fired`, so a client can distinguish "the Planner matched a keyword" from "the fallback classifier decided" on any given turn.

**Test count: 523 → 549 (+26, `test_planner_tool_fallback_classifier.py`).** Covers flag-off (classifier never called); Gate 1 (skips when the prior turn fired a tool, proceeds when it didn't, and `route()` itself populates the gate-1 state for the next turn with no external wiring); Gate 2 (skips on missing `memory_manager`/`runtime`); the parser (11 parametrized variants — case/whitespace-normalized valid names, `"none"`, empty string, prose, a disallowed name); shadow mode (classifier runs and logs, `RoutingPlan` identical to flag-off for the same input); and active mode (tool name sets `tools_to_call` + `tool_signal_source`, a parsed `none` still falls to P6, and — checked via a spy on `_classify_tool_fallback` rather than only asserting matching output — a genuine P3 keyword match short-circuits `route()` before the classifier method is ever reached). This count is order- and environment-sensitive in the full suite; see the open item below.

**Live shadow-mode verification (2026-07-04).** With the flag set to `shadow` against the real backend, the verbatim semantic-miss utterance from the diagnostic corpus above — "Go head and look up the weather in Oceanside CA." — correctly produced `parsed_tool='web_search'` in the log, on a turn where the Planner's own P1–P5 evaluation found no match (confirming both gates passed and the model call actually ran). On separate turns where P1–P5 matched deterministically, the classifier method was confirmed via log inspection to never be invoked at all — evidence the gate design itself is working, not just that its output happened to look right.

**Open item, blocking merge — order-dependent test failure, not this feature's own correctness. CLOSED 2026-07-05.** `test_implicit_hook_fires_on_plain_query` (`test_episodic_phase5.py`) passes in isolation but fails when the full suite runs with this branch's code present — confirmed via layer-by-layer bisection: clean HEAD → 2 failures / 523 passed (both pre-existing and network-dependent — see §14.6); adding this branch's classifier code with no other change → 3 failures / 522 passed, the extra one being this test. Not yet diagnosed to a root cause; suspected shared state leaking across tests, introduced somewhere in this branch's changes. This sits underneath the classifier's own "26/26 passing" figure — true in isolation for that one file, not true in the context of a full-suite run — and is a distinct finding from the clean shadow-mode verification above, not a contradiction of it. Must be resolved before this branch is considered mergeable. The full-suite pass/fail split is additionally noisy for an unrelated reason: two of the pre-existing failures (§14.6) only manifest when a live `localist-mcp` process happens to be bound to port 8003 with a real `LANGSEARCH_API_KEY` at test time, so a given run may show 1, 2, or 3 failures depending on ambient local state — the order-dependent one above is the only failure confirmed to track code, not environment.

**Root cause and closure, 2026-07-05.** Not a shared-state leak in this branch's own code, as suspected above — confirmed via log evidence, not inferred from correlation. `backend/.env` line 37 shipped `LOCALIST_TOOL_FALLBACK_CLASSIFIER=shadow`, contradicting its own "off by default" comment. `main.py`'s `load_dotenv()` runs at import time and sets this process-wide; because pytest imports every test module during collection before running any test, and `tests/test_main_task_chat_turns.py` imports `main`, the mere presence of that file in `tests/` poisoned `os.environ` for the entire suite — regardless of file execution order and regardless of shell state. Mechanism: the affected test's `rt.infer.side_effect` is a single-item list, on the documented assumption that `route()` makes zero `infer()` calls for its instruction (P5 is keyword-based). That instruction falls through P1–P5 to this section's classifier; with the flag resolving to non-`off`, `_classify_tool_fallback()` fires during `route()`, consumes the test's only `side_effect` value, and the real implicit-extraction call later in the turn hits `StopIteration` — confirmed live via `--log-cli-level=INFO`, where the classifier's `raw_output` was literally the string meant for the implicit-extraction call. This was also live in every real app run, not just tests: the classifier had been making an unaccounted-for `runtime.infer()` call on every P6-fallthrough turn in actual usage for however long that `.env` line predates this discovery.

Full-suite baseline: 566 passed / 1 failed (before either fix) → **567 passed / 0 failed** (after both fixes, confirmed under both unset and explicitly-active ambient conditions for this flag). This branch is no longer blocked by this item.

This is not a return to `shadow`; both flags are live together for trial purposes as of this update, which is a distinct state from either `off` or `shadow`.

### 15.2 lookup_request Resolved-Context Guard

**Root cause, documented first because it was the precondition for everything below.** Live traffic surfaced that `_priority3_tool()`'s DEBUG log, whenever the semantic gate fired, reported the global `best_group`/`best_score` argmax across all four `_SEARCH_INTENT_TEMPLATES` groups (§10.2) — even when a *different* group (or none, at the reported score) was the actual gating cause. Since only `explicit_search_action` and `lookup_request` gate routing, a log line could read `best_group='knowledge_request_open'` on a turn that actually fired via `lookup_request` independently clearing its own, separate, lower threshold — misattributing the cause to a non-gating group in the record. Fixed, logging-only: the log now names the specific threshold-crossing group(s) — computed the same way `semantic_triggered` already was (§10.2's gate check, now retained as a list instead of discarded inside `any()`) — instead of the argmax. `test_planner_phase3.py`: 110/110 before and after, confirmed by reverting just this change and re-running.

**What that fix made possible.** With the log now naming the actual gating group instead of the argmax, `diagnostics/score_referential_followup_lookup_request.py` (same live `Planner._semantic_search_intent()` methodology as every diagnostic in §10.4) scored 10 referential/follow-up phrases — "What do you make of all that?" (verbatim, live-confirmed 0.6158), "What do you think about that?", "Any thoughts on that?", "Is that concerning?", "What does that mean?", "So what's the deal there?", "Anything else notable in that?", "Can you unpack that a bit?", "Does that seem right to you?", "What's your take?" — against all four groups, alongside the three confirmed §10.4 Open Item 3 true positives for direct comparison. **9 of 10 cross the `lookup_request` 0.60 gate** (only "What's your take?" at 0.5955 falls short), while `explicit_search_action` stays nowhere close for the entire category (max 0.6247 vs. its 0.72 gate). The scores do not separate from real positives at any threshold: referential range 0.5955–0.7028 overlaps the true-positive range 0.649–0.778 on both sides — one confirmed true positive ("Can you look up their next-generation in-house Microsoft AI models?", 0.6585) scores lower than three of the nine referential false-fires. No threshold value fixes this; it needs a different signal.

**The guard.** `_priority3_tool()` now withholds `web_search` when `lookup_request` is the *only* group that cleared its gate (no literal `_WEB_SEARCH_KEYWORDS` match, and `explicit_search_action` did not independently clear its own gate) AND the prior turn already fired a tool (`self._last_turn_tools_fired` non-empty — the same state §15.1's Gate 1 already tracks). Rationale: a bare `lookup_request` collision immediately after a tool call reads as "referring back to that result," not "asking for something new." A literal keyword match or an independent `explicit_search_action` crossing are untouched under all conditions — this targets the `lookup_request` collision specifically, since it is the only group the diagnostic implicated. `file_op`, `url_fetch`, and P3c graph-query signals are entirely unaffected.

**Three-stage flag — `LOCALIST_LOOKUP_REQUEST_RESOLVED_CONTEXT_GUARD`, default `off`.** Same off/shadow/active pattern as §15.1: `off` is a no-op; `shadow` evaluates and logs `would_suppress` (naming the instruction, `last_turn_tools_fired`, and the `lookup_request` score/threshold) but still adds `web_search`; `active` actually suppresses, letting the turn fall through toward P3b/P4/P5/(§15.1's classifier, if also active)/P6.

**Test count: 16 new (`test_planner_lookup_request_resolved_context_guard.py`).** Covers flag-off (zero change, including for the exact referential phrases above with a prior tool fired); the literal-keyword and independent-ESA cases (untouched in every mode, including active, with a prior tool fired); active-mode suppression using the real referential phrases as fixtures, not synthetic stand-ins (both `_priority3_tool()` returning `None` directly and an end-to-end `route()` call landing on Priority 6); the no-prior-tool case (guard does not suppress — this is §10's original gate behavior and must remain intact); and shadow mode producing a `RoutingPlan` identical to flag-off for the same instruction/history pair. Full suite: 522 passed / 3 failed both before and after this guard's addition — the same 2 pre-existing network-dependent failures plus §15.1's order-dependent regression, none attributable to this guard, confirmed by reverting only the guard's code and re-running the full suite (the count did not change).

**Open item, blocking any use beyond `off` — the guard cannot distinguish a referential no-op from a genuinely new, more specific topic. OPEN — re-scoped 2026-07-05, live-trial data collection ongoing, NOT closed.** The guard's rule was designed and tested against the diagnostic's referential-phrase corpus, all of which share zero topic content with whatever preceded them — pure follow-up language pointing at something already in context. Live traffic (2026-07-04, shadow-adjacent — the guard was not active for this turn, so no suppression occurred and the search correctly fired) surfaced a case the diagnostic corpus never tested: a prior turn searched "What's the latest Apple news?" (general query, 3 mixed results); the immediate follow-up, "What do you think of Apple's new Siri AI announcement?", scored `lookup_request=0.699`, fired alone, with `last_turn_tools_fired=['web_search']` already true — satisfying both of the guard's suppression conditions exactly as a pure referential phrase would, despite being a legitimate, more specific lookup that deserves its own grounded search rather than being answered from the prior turn's general results already in context. The guard's two conditions (group-alone gate crossing + prior-turn tool state) carry no information about topic overlap or specificity between the new instruction and what the prior tool call actually covered; closing this gap requires some measure of semantic overlap between the two, out of scope for what this guard does today. **The guard must remain `off`** until either a topic-overlap signal is added or this trade-off is consciously accepted and documented; the test suite should gain explicit "must NOT suppress" cases using this exact utterance and similar specific-new-topic-after-tool-call variants before shadow testing resumes on this guard.

**Update 2026-07-05 — combined live trial with §15.1's classifier, still OPEN, re-scoped rather than closed.** With §15.1's blocking item resolved, both `LOCALIST_TOOL_FALLBACK_CLASSIFIER` and this guard's flag were set to `active` together against real conversational traffic, specifically to gather live frequency data on the referential no-op problem before committing to a fix direction — not a decision to accept the guard's current behavior. First live confirmation (previously only code-traced) that the two features' shared-state coupling holds outside test mocks: `self._last_turn_tools_fired` is fixed for the whole `route()` call and only updated afterward via `record_tools_fired()`, so on every turn where this guard's suppression condition held, §15.1's Gate 1 independently saw the same non-empty state and skipped itself — confirmed via paired log lines on 3 separate turns. The classifier cannot re-add `web_search` on a turn this guard just suppressed it on; this was a real (if code-verified) concern going into the trial and is now empirically confirmed safe.

Guard-specific trial data (n=3, at the `lookup_request`-alone boundary): one genuine referential follow-up ("What do you make of that?" after a gold-price search, `lookup_request=0.629`, gate cleared) was correctly suppressed — confirmed true positive. Two attempts at adjacent-but-different-topic phrasing — "What do you think of Gold vs Silver in a portfolio?" after a silver-price search (`lookup_request=0.572`) and "What do you make of the new Siri AI?" after a general Apple-news search (`lookup_request=0.596`) — never reached the guard's suppression-vs-decline branch at all: both fell just under the 0.60 `lookup_request` gate. This reveals a separate, pre-existing semantic-gate coverage gap on comparative/advisory phrasing (§10.4), not a defect in this guard. The second case is also, on inspection, not a clean topic-adjacency test: the prior search result already contained the follow-up's specific content directly ("Apple unveiled the next generation of Apple Intelligence and Siri AI during WWDC 26"), so suppression would have been defensible there regardless — worth correcting for the record, since this exact scenario had earlier been discussed as illustrative before it was actually run live.

This does not close or deprioritize the referential no-op problem. It confirms the failure mode hasn't yet been observed live, and that real adjacent-but-different-topic phrasing may cluster closer to the `lookup_request` gate boundary than the original synthetic diagnostic's hand-picked examples did — the diagnostic's 9/10-cross-gate finding above is unchanged and not in question; what's newly in question is how often wrong-topic phrasing also clears that same gate in real usage, and the trial so far suggests less often than assumed. **n=3 is not enough volume to revise this item's priority**, per the trial record. The item remains OPEN and re-scoped: before investing in a sharper signal (embedding similarity between the current instruction and the prior tool query/result; narrowing the guard's scope to specific prior-tool types or a shorter turn window — both sketched, neither built) more live trial volume is needed to establish whether this failure mode manifests at a rate worth engineering against, versus being a smaller real-world risk than the original synthetic diagnostic suggested. Both flags are left `active` together for continued organic data collection during normal use — a live-trial state, distinct from both `shadow` and from this guard remaining `off`.

**Update 2026-07-08 — guard removed. Item CLOSED (by removal, not by a sharper signal).**

**Originally:** the guard suppressed `web_search` whenever any tool was scheduled on the immediately preceding turn, regardless of that tool's success or failure, added as a fix for an over-eager web-search trigger.

**Decision (2026-07-08):** a live false negative showed the guard blocking a legitimate consecutive tool request — a `lookup_request`-classified turn following a FAILED `url_fetch` (a 403). A read-only diagnostic pass confirmed the guard was a pure boolean on prior-turn tool-scheduling state with no success/failure awareness at all — not a threshold-calibration issue, since the guard never had a numeric threshold of its own. The decision was to remove the suppression entirely rather than refine it, on the principle that users should not be blocked from legitimate back-to-back tool use.

**Fix:** removed the suppression block, `_resolved_context_guard_mode()`, `_LOOKUP_GUARD_ENV_VAR`, and all mode handling from `planner.py` (128 lines → net 8 insertions / 120 deletions). Removed `LOCALIST_LOOKUP_REQUEST_RESOLVED_CONTEXT_GUARD` from `backend/.env`. `web_search` now fires purely off the `explicit_search_action` (0.72) / `lookup_request` (0.60) semantic gates, with no prior-turn tool-state involved. `record_tools_fired()`/`_last_turn_tools_fired` were retained unchanged — confirmed independent of this guard, since both are still used by §15.1's P6-fallthrough tool-fallback classifier's Gate 1.

**Live-verified:** reproduced the original failing transcript against the fixed `Planner` — the previously-suppressed turn and its verbatim repeat both now correctly fire `web_search`, with no divergence between them.

**Test suite:** 593 → 576 passed (delta of 17 matches the deleted `test_planner_lookup_request_resolved_context_guard.py` file exactly, confirmed by direct count of its collected tests); the same 2 pre-existing, unrelated, network-dependent failures (§14.6) unchanged both before and after.
